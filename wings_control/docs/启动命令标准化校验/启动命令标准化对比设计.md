# 启动命令标准化对比设计

## 1. 文档信息

- 文档状态：MVP 已实现，设计持续演进
- 适用项目：`wings_control`
- 首期范围：vLLM / vLLM-Ascend 的 0 Day 模型启动命令
- 后续范围：SGLang、MindIE、单机、多机、PD 分离等启动形态
- 核心目标：在相同模型、硬件、引擎和特性输入下，验证 Wings 生成的最终启动脚本与已批准的标准语义一致

当前 MVP 实现：

- `startup_command_standard.py`：提供 `draft`、`compare`、`verify` 三个子命令；
- `test_startup_command_standard.py`：覆盖非标准输入整理、语义比较、错误识别和生产链路验证；
- `examples/qwen35-27b-910b.scenario.json`：复用现有 DAY0 资源的场景示例；
- `README.md`：输入、输出、退出码和执行示例。

## 2. 背景

当前模型适配资料分散在测试结果 Excel、DAY0 示例脚本、优化命令、Wings dry-run 产物和项目配置中。不同来源承担的职责不同：

- 测试结果 Excel 记录测试场景、原生命令、优化命令和测试状态；
- DAY0 `.sh` 文件记录特定模型与硬件组合的示例或交付脚本；
- Wings 配置、白名单和适配器共同决定实际生成的最终命令；
- dry-run 产物反映某次代码和配置状态下的生成结果。

这些资料不能直接互相替代。尤其不能把 Wings 当前生成结果本身当作标准，否则生成逻辑和期望值同时发生错误时仍可能“自证通过”。因此需要建立一个独立、可审核、可追溯的标准层。

本设计采用以下闭环：

```text
已验证可用命令 / 官方 recipe / 测试结果
                    ↓
              标准候选提取
                    ↓
             人工确认关键语义
                    ↓
          版本化的场景标准清单
                    ↓
相同场景输入 → Wings 生产链路 → 最终启动脚本
                    ↓
          解析、归一化、语义对比
                    ↓
          PASS / FAIL / REVIEW 报告
```

## 3. 目标与非目标

### 3.1 目标

1. 接收一条简单、已验证可用的启动命令，整理为可审核的标准场景草案。
2. 使用与生产一致的 Wings 命令生成链路生成实际启动脚本。
3. 忽略参数顺序、引号和换行等无语义差异，按字段比较真实含义。
4. 同时支持“必须出现”“必须等于”“禁止出现”和“跟随输入”的规则。
5. 对模型路径、端口、IP 等动态值进行受控比较，不通过全局忽略掩盖问题。
6. 输出可供开发、模型适配和测试人员共同审核的字段级报告。
7. 将已批准场景接入 CI，防止配置、白名单和命令生成逻辑回归。

### 3.2 非目标

1. 静态命令一致性不替代真实硬件启动和推理验证。
2. 不自动认定任意 Excel 单元格或历史脚本为正确标准。
3. 不允许测试失败后自动覆盖标准快照。
4. 不在第一阶段一次性覆盖所有 0 Day 场景。
5. 不通过复制一套新的命令生成逻辑来模拟 Wings 行为。

## 4. 标准的基本原则

### 4.1 标准必须独立于被测实现

标准描述“该场景应该具备的启动语义”，Wings 代码负责生成“实际结果”。标准不得从实际结果自动反向更新。

### 4.2 一个标准场景只描述一个确定组合

以下任一维度不同，都应视为不同场景：

- 模型及量化变体；
- 硬件系列、显存规格和卡数；
- 单机、多机、DP、Ray 或 PD 拓扑；
- 推理引擎和引擎版本；
- MTP、MemCache、稀疏、EP 等特性组合；
- 影响启动语义的镜像或加速包版本。

推荐场景标识：

```text
<模型>-<精度>-<硬件>-<卡数>-<拓扑>-<引擎版本>
```

示例：

```text
qwen36-27b-w8a8-910c-2card-single-vllm-ascend-021
```

### 4.3 先比较语义，再保留快照

标准的核心是结构化规则。规范化命令快照用于审阅整体变化，但不能代替语义断言。

### 4.4 正向规则与负向规则同等重要

除了检查参数是否存在，还必须检查不适用于该场景的参数是否被错误注入。例如：

- 910B 场景按批准策略禁止 KV offload 时，不得出现 MemCache 前置片段和 `--kv-transfer-config`；
- 非 MoE 场景不得错误启用 expert parallel；
- 未批准稀疏特性的场景不得出现 `--sparse-config`；
- 不得出现重复或互相冲突的 CLI 参数。

## 5. 真值来源与状态管理

### 5.1 建议的来源优先级

1. 在目标硬件和指定引擎版本上验证通过、且由责任人确认的完整命令；
2. 官方 recipe 或模型适配团队正式交付命令；
3. 测试结果表中已确认通过的优化命令；
4. DAY0 同场景完整脚本；
5. 尚未实测的设计参数，仅能作为候选，不得直接作为强制标准。

Wings 当前生成命令属于被测对象，不作为独立真值来源。

### 5.2 标准状态

每个场景必须具有以下状态之一：

- `candidate`：已从命令提取，尚未完成业务确认；
- `provisional`：规则基本明确，但仍有未解决问题；
- `accepted`：已完成来源、规则和测试证据确认，可阻断 CI；
- `deprecated`：旧引擎、旧镜像或旧策略场景，仅保留历史追溯；
- `rejected`：确认不可用或不再采用的候选。

只有 `accepted` 场景参与强制门禁。`candidate` 和 `provisional` 仅生成报告。

### 5.3 冲突处理

当 Excel、DAY0 脚本、官方 recipe 与 Wings 结果不一致时，不按“多数来源”自动决定。必须生成冲突清单，并由相应责任人裁定：

- 模型适配负责人确认原生命令和模型特性；
- Wings/Control 负责人确认场景输入映射和生成行为；
- 测试负责人确认真实硬件验证结果；
- 标准维护人记录最终结论、依据和版本。

## 6. 用户需要提供的最小输入

用户可以直接提供一条或一段已验证命令，不需要预先整理成 YAML。最小信息如下：

```text
模型名称：
量化或精度：
硬件类型：
卡数：
单机/多机：
推理引擎及版本：
镜像或加速包版本（如已知）：
启动命令：
是否在真实环境启动成功：
测试证据或责任人（如有）：
```

推荐提供完整脚本，至少包括：

1. 与模型相关的 `export` 环境变量；
2. MemCache、Ray、分布式等前置命令；
3. 最终 `vllm serve` 或 `python3 -m ...` 引擎命令；
4. 模型路径、端口和网络变量的含义；
5. 实际测试所用引擎和镜像版本。

如果只提供核心引擎命令，系统仍可建立 CLI 标准，但环境变量、前置服务和完整拓扑规则必须标记为“未覆盖”，不得伪装为已验证。

## 7. 从简单命令生成标准草案

### 7.1 处理步骤

```text
接收原始命令
  → Shell 分段与安全解析
  → 提取 export、前置命令和核心引擎命令
  → 解析 CLI 和嵌套 JSON
  → 识别固定值、输入关联值和运行时动态值
  → 推导建议的 required / forbidden 规则
  → 标记无法从单条命令确定的问题
  → 输出 candidate 标准及待确认清单
```

### 7.2 不应自动推断的内容

以下结论不能仅凭“命令中没有出现”自动确定：

- 某参数是否明确禁止；
- 某特性是关闭、使用默认值，还是命令遗漏；
- 某环境变量是否由镜像内脚本隐式提供；
- 命令是否适用于其他硬件或其他引擎版本；
- 实际运行成功是否代表性能和特性均达到验收要求。

这些内容应进入待确认清单。

## 8. 标准数据模型

推荐将最终标准保存为可代码审查的 YAML 或 JSON。Excel 保留为测试来源和证据，不直接承担 CI 真值。

示例：

```yaml
schema_version: 1
id: qwen36-27b-w8a8-910c-2card-single-vllm-ascend-021
status: accepted

identity:
  model: Eco-Tech/Qwen3.6-27B-w8a8

    - Qwen3.6-27B-W8A8

  engine_version: "0.21"

  enable_speculative_decode: true
  enable_kv_offload: true
  enable_sparse: false

  entrypoint:
    kind: python_module

  cli:
    exact:
      tensor-parallel-size: 2

      - trust-remote-code
      - enable-auto-tool-choice

      - sparse-config

  approved_at: "待填写"
```

### 8.1 标准中必须记录的内容

- 场景唯一标识；
- 模型规范名和允许的别名；
- 硬件、卡数、拓扑；
- 引擎和版本；
- 输入特性开关；
- 必须精确匹配的参数；
- 必须存在的布尔参数和环境变量；
- 禁止出现的参数或 JSON 路径；
- 动态字段规则；
- 来源、实测状态和审批信息。

## 9. Wings 实际命令生成

### 9.1 必须复用生产链路

校验器必须从标准场景构造与真实启动一致的输入，并调用项目生产链路：

```text
parse_launch_args()
  → derive_port_plan()
  → build_launcher_plan()
      → 配置加载和合并
      → 硬件识别
      → 特性白名单和有效开关解析
      → engine_manager.start_engine_service()
      → adapter.build_start_script()
      → 完整 start_command.sh 组装
```

不得另写一套命令拼装器模拟生产行为。校验代码只负责：

- 构造确定的场景输入；
- 隔离环境变量；
- 提供稳定的硬件和模型配置夹具；
- 调用生产入口；
- 解析最终输出；
- 与独立标准对比。

### 9.2 校验范围

最终比较对象应包含三层：

1. 合并后的有效配置和生效特性；
2. 完整 `start_command.sh` 中的环境变量和前置片段；
3. 最终实际执行的引擎命令。

只比较第三层可能漏掉 MemCache 服务、加速包安装、网络环境变量和分布式初始化错误。

## 10. 命令解析与归一化

### 10.1 解析结果

原始脚本应解析为以下结构：

```text
NormalizedScript
├── environment: Map<变量名, 最终值>
├── duplicate_environment: List<重复变量>
├── pre_commands: List<规范化前置命令>
├── engine_commands: List<NormalizedCommand>
│   ├── entrypoint
│   ├── positional_args
│   ├── flags
│   ├── duplicate_flags
│   └── json_flags
└── topology
    ├── node_count
    ├── node_rank
    ├── tp
    └── dp
```

### 10.2 允许归一化的差异

- CLI 参数顺序；
- JSON 对象键顺序；
- 单引号和双引号；
- Shell 续行和空白；
- 不影响语义的 `exec` 包装；
- 注释、echo 日志和日志重定向；
- 数值字符串与对应数值类型；
- 经标准明确批准的等价引擎入口。

### 10.3 不允许被归一化掉的差异

- 重复 CLI 参数；
- 同一环境变量被多次赋予不同值；
- 参数缺失、多余或值变化；
- JSON 内部字段缺失或类型变化；
- 前置服务、角色或端口变化；
- 单机和多机拓扑变化；
- MTP、EP、量化、offload 等关键特性变化。

### 10.4 动态字段规则

动态字段必须逐字段声明，禁止使用“忽略所有路径/IP/端口”的宽泛规则。

推荐规则：

- `equals_input`：实际值必须等于场景输入；
- `matches_runtime_policy`：实际值必须符合项目运行时策略；
- `normalized_path`：只允许路径根变化，模型目录名必须一致；
- `allocated_port`：实际端口必须等于端口规划结果；
- `any_nonempty`：只要求存在，适用于临时生成文件路径；
- `ignored_with_reason`：确需忽略时必须记录理由。

## 11. 语义比较规则

### 11.1 比较类别

| 类别 | 含义 | 示例 |
|---|---|---|
| `exact` | 必须存在且值完全一致 | TP、量化方式、tool parser |
| `present` | 必须出现，值可由动态规则检查 | `--trust-remote-code` |
| `absent` | 必须不存在 | 910B 禁止的 offload 参数 |
| `json` | 递归比较 JSON 字段 | `--speculative-config` |
| `dynamic` | 必须满足输入或运行时关系 | 模型路径、端口、主机地址 |
| `set_equal` | 集合相等，忽略顺序 | 生效特性集合 |
| `snapshot` | 仅用于整体审阅 | 规范化完整命令 |

### 11.2 建议的关键断言

- 模型和 served model name；
- TP、DP、EP 和节点拓扑；
- 上下文、并发和批处理参数；
- 显存利用率和 seed；
- 量化方式；
- MTP method、token 数和 draft model；
- reasoning parser、tool parser 和自动工具调用；
- compilation config、additional config；
- KV connector、role、backend 和前置服务；
- 稀疏参数；
- 关键环境变量；
- 生效特性集合与允许特性集合；
- 重复参数和相互冲突参数。

## 12. 对比结果与问题分类

### 12.1 结果状态

- `PASS`：所有强制规则通过；
- `FAIL`：与 accepted 标准不一致；
- `REVIEW`：标准处于 candidate/provisional，或存在未裁定字段；
- `ERROR`：场景输入、模型夹具或生成链路自身失败。

### 12.2 差异报告示例

```text
FAIL qwen36-27b-w8a8-910c-2card-single-vllm-ascend-021

[值不一致] speculative-config.num_speculative_tokens
  标准值: 3
  实际值: 1

[缺失] kv-transfer-config.kv_connector
  标准值: AscendStoreConnector
  实际值: null

[禁止项] sparse-config
  标准值: 不允许出现
  实际值: 已出现

[动态字段通过] port
  输入值: 18000
  实际值: 18000
```

### 12.3 差异处理

每个差异只能进入以下处置之一：

1. Wings 生成错误：修改配置、白名单或命令生成逻辑；
2. 场景输入错误：修正标准场景的输入夹具；
3. 标准过期：提供新测试证据并走标准更新审批；
4. 动态差异：补充精确的动态规则；
5. 来源冲突：保持 REVIEW，等待责任人裁定。

禁止为了让 CI 通过直接用实际输出覆盖标准。

## 13. 标准建立、对比和更新流程

### 13.1 新建标准

1. 用户提交已验证命令和最小场景信息；
2. 工具提取 candidate 标准；
3. 输出固定值、动态值、建议禁止项和待确认项；
4. 模型适配负责人确认模型和引擎语义；
5. Wings 负责人确认场景输入映射；
6. 测试负责人关联真实测试证据；
7. 标准状态改为 accepted；
8. 提交标准文件和规范化快照。

### 13.2 执行对比

1. 读取 accepted/candidate 场景；
2. 清理并固定影响生成结果的环境变量；
3. 构建模型配置和硬件信息夹具；
4. 调用 `build_launcher_plan()`；
5. 保存原始脚本、核心命令和合并配置；
6. 解析并归一化实际结果；
7. 执行语义规则、禁止项和重复项检查；
8. 生成 JSON 和 Markdown 报告；
9. accepted 失败时返回非零退出码。

### 13.3 更新标准

更新必须包含：

- 变更的场景和字段；
- 变更原因；
- 新命令或新官方 recipe；
- 引擎、镜像和硬件版本；
- 新的测试证据；
- 审批人；
- 对旧场景是替换、并存还是废弃的说明。

标准更新与生产代码修改应在评审中分别展示，避免期望值变化掩盖实现回归。

## 14. Excel 与 DAY0 资料接入

### 14.1 Excel 的职责

`AI Infra工作台_0 Day测试结果.xlsx` 用于建立候选场景清单和追溯测试证据。导入时应：

- 去除模型名称尾部空格；
- 合并或显式记录模型别名；
- 以模型、精度、硬件、卡数、拓扑、引擎版本去重；
- 区分基线、优化和 Wings 命令；
- 保留原始行号；
- 根据测试状态生成 candidate/provisional 建议；
- 不把空白命令解释为“该参数不需要”。

### 14.2 DAY0 脚本的职责

DAY0 脚本可用于：

- 补充 Excel 中被包装脚本隐藏的完整参数；
- 解析前置服务和环境变量；
- 对照同模型不同硬件的差异；
- 作为标准来源证据之一。

脚本必须经过场景归属和实测状态确认后才能成为 accepted 标准来源。

## 15. 现有能力复用与需要补齐的部分

### 15.1 可直接复用

现有 `DAY0/verify_qwen_day0_dry_run.py` 已具备以下原型能力：

- 构造模型和硬件夹具；
- 调用 `build_launcher_plan()` 走生产链路；
- 提取最终引擎命令；
- 解析 CLI 和 JSON 参数；
- 比较 expected 与 actual；
- 校验 MemCache 端口和前置片段；
- 输出 `comparison.json` 和 `comparison.md`。

后续应优先演进并复用这些能力，避免创建一套平行校验器。

### 15.2 当前需要补齐

1. 将硬编码的 Qwen 场景和 expected 数据迁移到独立标准文件；
2. 建立标准 schema 和版本；
3. 支持完整 Shell 解析、重复参数检测和多命令拓扑；
4. 增加 `candidate/provisional/accepted` 状态；
5. 增加来源、审批和测试证据；
6. 将动态字段从简单忽略改为关系校验；
7. 支持禁止项和未知参数策略；
8. 支持除 Qwen 外的模型与 NVIDIA 场景；
9. 修复当前来源工作簿审计，使标准数据真正来自批准后的场景清单；
10. 将报告接入 CI。

当前 `qwen模型收编.xlsx` 只有表头，不能作为可执行的标准来源；包含 11 个 Qwen 优化场景的 `qwen优化参数适配基准.xlsx` 也不能只检查“文件存在”，而应在标准迁移阶段读取、审核并转成版本化标准清单。

## 16. CI 与发布门禁

### 16.1 每次提交

建议每次提交执行：

- 标准 schema 校验；
- accepted 场景生产链路生成；
- 语义对比；
- 禁止项检查；
- 重复参数检查；
- 规范化快照差异展示。

以下文件变化时必须执行全量场景：

- 默认配置；
- 智能特性白名单；
- 配置加载和合并逻辑；
- 硬件识别逻辑；
- 引擎适配器；
- Wings 启动编排；
- 标准场景文件。

### 16.2 夜间或发布前

- Shell 语法检查；
- 对应镜像内的引擎参数解析检查；
- 选定硬件上的服务 ready 检查；
- 最小推理请求；
- MTP、MemCache、EP 等特性实际生效检查。

### 16.3 代码评审要求

- 标准变化和实际命令变化必须分别展示；
- accepted 标准更新需要指定责任人审核；
- 不允许无来源地更新快照；
- 新参数必须说明属于 required、forbidden、dynamic 或非关键快照字段中的哪一类。

## 17. 分阶段实施建议

### 阶段一：Qwen 11 场景试点

1. 选定当前 11 个 Qwen 正式场景；
2. 整理 Excel、DAY0 脚本和实际生成结果；
3. 裁定 910B offload、Qwen3.6 tool parser、BF16 量化和 MTP token 等冲突；
4. 建立第一版标准 schema；
5. 改造现有 Qwen dry-run 校验器读取外部标准；
6. 生成语义报告但暂不阻断 CI；
7. 完成责任人确认后将场景改为 accepted。

### 阶段二：NVIDIA、DeepSeek、Kimi、GLM

- 接入 H20/L20 的 9 个场景；
- 接入 DeepSeek V4-Flash 的 Ascend 与 NVIDIA 变体；
- 接入 Kimi K2.6/K2.7；
- 接入 GLM 单机和多机场景；
- 扩展多命令和分布式拓扑解析。

### 阶段三：项目级门禁

- accepted 场景全量接入 CI；
- 建立标准变更审批规则；
- 在发布流水线增加真实硬件抽样；
- 形成标准覆盖率、失败率和待确认项统计。

## 18. 验收标准

首期功能完成应满足：

1. 用户提供一条可用命令后，可生成结构化 candidate 标准；
2. 标准草案明确列出固定值、动态值、禁止项建议和待确认项；
3. 校验器通过生产链路生成完整启动脚本；
4. 参数顺序、引号和 JSON 键顺序变化不造成误报；
5. TP、MTP、量化、EP、offload 等关键差异能够准确失败；
6. 重复参数和相互冲突参数能够准确失败；
7. 报告包含标准值、实际值、来源和处置建议；
8. candidate/provisional 不阻断 CI，accepted 失败阻断 CI；
9. 标准不能通过普通对比流程自动覆盖；
10. 至少 11 个 Qwen 场景完成标准化对比。

## 19. 用户在首期需要确认的事项

为了将 Qwen 试点从设计推进到 accepted 标准，需要确认：

1. 首期是否固定为现有 11 个 Qwen 场景；
2. 每个场景以哪条实测命令或官方 recipe 为真值；
3. 910B 是否统一禁止 MemCache/offload；
4. Qwen3.6 的 tool parser 采用 `hermes` 还是 `qwen3_coder`；
5. Qwen3.6 BF16 是否需要 `--quantization ascend`；
6. 每个场景的 MTP token 数；
7. 哪些模型路径、端口和 IP 属于动态字段；
8. accepted 标准的审批人和测试证据格式。

## 20. 保证边界

该机制能够保证：

- 在确定输入和固定代码、配置版本下，Wings 生成命令符合批准的语义标准；
- 配置、白名单或命令生成逻辑变化时能够发现命令回归；
- 标准变化有来源、有审批、可追溯。

该机制不能仅凭静态比较保证：

- 引擎镜像一定支持全部参数；
- 硬件、驱动、网络和模型权重状态正确；
- 服务一定成功启动；
- 性能一定达到目标。

因此最终发布仍需要真实硬件的启动、ready、最小推理和特性生效验证。
