# AI Infra 0 Day 白名单差异场景分析

分析日期：2026-07-16

源数据：`AI Infra工作台_0 Day测试结果_收编视图.xlsx`

白名单：`../../config/smart_feature_whitelist.json`

配套抽取表：`AI Infra工作台_0 Day测试结果_白名单差异场景及优化参数.xlsx`

## 1. 分析口径

- 模型名称取源表 D 列，特性取 M 列，芯片取 C 列，优化参数取 Q 列。
- spec 包含 MTP、Eagle、suffix、DFlash。
- sparse 包含 IndexCache、KVCache FP8；表格中的 FP8 统一归为 sparse。
- offload 区分具体后端：Native、LMCache、MemCache、MoonCake Store。
- PD 分离不纳入本次差异判断。
- 暂不分析 GLM5.1/GLM5.2 与 Ascend 的组合场景，即源表行 12、33、45、49。
- 910B 场景不要求 MemCache，删除源表行 13、14、17 的 offload 需求。
- 源表行 26 的 Qwen-AgentWorld + Pro5000 不要求 LMCache，仅保留 suffix/spec。
- H20 同时检查 `h20-96` 和 `h20-141`，不能只以其中一种卡型代表整个 H20 场景。
- H20 三个 GLM 场景中，D 列的 `FP8` 是精度/场景标签，不属于开源模型身份。实际模型名以 Q 列启动命令为准，分别为 `zai-org/GLM-4.7`、`zai-org/GLM-5.1`、`zai-org/GLM-5`。
- DeepSeek-V4-Flash 的本轮 NVIDIA 差异只来自源表行 27 的 H20 场景；`rtxpro5000-72` 是 23.6.0 历史白名单场景，不在本轮差异 Excel 中，因此不能借本轮补丁新增或增强 Pro5000-72 sparse 策略。
- 白名单按 engine 精确匹配、`model_name + model_path` 子串匹配、card token 匹配，并应用 `exclude_name_tokens` 负向过滤。

## 2. 汇总结论

本轮共纳入 38 个有效场景：

- 27 个场景未发现差异。
- 11 个场景存在功能差异或模型身份问题。
- 差异名单共 14 项：12 项确定差异、1 项 H20 型号条件差异、1 项当前功能可命中但模型身份不规范的结构风险。
- H20 + GLM 三个场景均已纳入：源表行 28 `GLM-4.7-FP8`、行 29 `GLM-5.1-FP8`、行 30 `GLM-5-FP8`。
- GLM-4.7 按正确基础名匹配后，MTP token=1 和 Native offload 已覆盖两种 H20，真实功能缺口只剩 KVCache FP8 sparse 行。
- GLM-5.1 按正确基础名匹配后，MTP token=3、IndexCache topk=4、Native offload 已覆盖两种 H20；当前问题来自旧 FP8 名称、负向排除和旧专用行之间的冲突。
- GLM-5 当前三特性均可命中，但 `GLM-5-FP8` 仍是错误模型身份，当前依赖宽子串匹配掩盖了该问题。
- 除三条 H20 GLM 外，行 47 `Kimi-K2.7-Code + 910C` 还存在源表身份与仓库现有 `-w4a8` 身份证据冲突；当前漏匹配是确定的，但应先确认实际权重身份，再决定修正源表还是新增白名单行。

问题场景为源表行 8、9、10、11、15、27、28、29、30、32、47。

## 4. H20 + GLM 专项分析

| 源表行 | D 列展示名 | 实际开源模型身份 | 当前白名单表现 | 正确目标 |
|---|---|---|---|---|
| 28 | `GLM-4.7-FP8` | `zai-org/GLM-4.7` | 使用展示名时：H20-96 三特性全 miss；H20-141 命中旧 23.6.0 行，得到 MTP3、sparse、LMCache。使用正确基础名时：两种 H20 均得到 MTP1、Native，但无 sparse | MTP token=1、KVCache FP8、Native offload，两种 H20 均覆盖 |
| 29 | `GLM-5.1-FP8` | `zai-org/GLM-5.1` | 使用展示名时：H20-96 三特性全 miss；H20-141 只命中旧 sparse 行。使用正确基础名时三特性均正确 | MTP token=3、IndexCache topk=4、Native offload，两种 H20 均覆盖 |
| 30 | `GLM-5-FP8` | `zai-org/GLM-5` | 展示名和正确基础名当前都能命中基础行，因为通用 token `glm-5` 未排除 FP8；功能正确但身份不规范 | MTP token=3、IndexCache topk=4、Native offload，两种 H20 均覆盖 |

根因不是 matcher 无法处理正确模型名，而是当前白名单同时保留了两套互相竞争的策略：

1. `vllm-0.23` 基础名规则使用 `zai-org/GLM-*`，覆盖 `h20-96` 和 `h20-141`。
2. 旧 `23.6.0` 规则使用 `ZhipuAI/GLM-*-FP8`，只覆盖 H20-141，且 GLM-4.7 的 MTP token 和 offload backend 与当前 Day0 要求不同。
3. GLM-4.7、GLM-5.1 基础名规则又通过 `exclude_name_tokens=["fp8"]` 排除错误展示名，因此即使 `model_path` 已是正确基础路径，只要 `model_name` 仍含 FP8，基础行仍不会命中。

## 5. 其他类似情况检查

### 5.1 Kimi 身份证据冲突

源表行 47 要求 MemCache。白名单已有 `vllm_ascend + 910c + memcache` 行，但 name token 是 `kimi-k2.7-code-w4a8`；源表 D 列是 `Kimi-K2.7-Code`，P/Q 启动路径也不带 `-w4a8`。匹配器不会自动补量化后缀，因此当前行确定无法命中。

但仓库中的 Kimi DAY0 设计、默认配置、模型识别及测试均使用 `Kimi-K2.7-Code-w4a8`。因此不能仅为通过源表检查就把白名单 token 放宽成父名称：

1. 如果实际权重是 W4A8，应修正源表 D/P/Q 或上游模型身份，继续保留精确 `-w4a8` 白名单。
2. 如果实际权重确实是不带量化后缀的独立模型，应新增一条精确模型策略，并同步该模型在默认配置、模型识别、测试和文档中的身份；不能让父 token 同时代表两个权重变体。

### 5.2 已排除范围内的潜在问题

源表行 49 的 D 列是 `GLM-5.2-w4a8`，P 列实际启动路径和白名单均是 `GLM-5.2-w8a8`。这是源表内部模型身份不一致，但属于本轮明确排除的 GLM5.2 + Ascend 场景，不计入当前差异统计。

### 5.3 DeepSeek Pro5000-72 不是本轮新增依据

白名单历史上已有 `DeepSeek-V4-Flash + rtxpro5000-72` 的 spec/sparse 覆盖，其中 sparse 行来自 23.6.0 历史矩阵，不是本轮 Excel 差异。源表本轮出现的 DeepSeek-V4-Flash NVIDIA 差异是行 27 `G8600+H20 * 8`，应只补齐 `h20-96` 和 `h20-141` 的 spec/sparse/offload 缺口。

因此，当前实现应避免把 H20 行的 IndexCache 结论外推到 Pro5000-72；如果 Pro5000-72 以后需要调整，必须有对应源表行、启动参数或独立验证记录支撑。

### 5.4 可由实际路径补偿，不构成当前漏匹配

- 行 11、32：D 列是 `DeepSeek-V4-Flash-w8a8`，实际路径带 `-mtp`，可命中白名单的 `deepseek-v4-flash-w8a8-mtp`。
- 行 21：D 列是 `MiniMax-M3`，Q 列实际路径是 `MiniMax-M3-MXFP8`，可命中对应白名单行。
- 行 43：D 列是 `DeepSeek-V4-Pro-w4a8`，实际路径带 `-mtp`，可命中对应白名单行。

### 5.5 其他宽匹配结构风险

- `Qwen3.5-397B-A17B-w8a8-mtp + 910C` 当前会被父 token `qwen3.5-397b-a17b` 命中，因此功能表面正常；但该父行也会覆盖量化/MTP 子型号。应为 910B、910C 分别增加准确 `-w8a8-mtp` 行，并让父行排除该后缀，避免不同卡型 token 数被宽匹配掩盖。
- `GLM-5` 通用 token 当前只排除 `glm-5.1`，仍可能命中 `GLM-5.2` 或错误的 `GLM-5-FP8` 展示名。基础行应同时隔离 `glm-5.1`、`glm-5.2` 和身份字段中的 `fp8`；各版本使用准确模型行。
- 旧 `ZhipuAI/GLM-4.7-FP8`、`ZhipuAI/GLM-5.1-FP8` 不是当前基础模型的合法别名，而是带有不同 token/backend 的旧策略。不能继续与 `zai-org/GLM-*` 基础行并存竞争，应从当前生效白名单移除或放入不参与匹配的历史记录。

除上述情况外，没有再发现新的“白名单已有对应能力，但 D、P、Q 中实际模型身份均无法命中”的场景。

## 7. 最小补丁实施口径

本轮实现不再重构 `config_loader`、`wings_entry`、MemCache 或 offload 后端选择链路；这些路径沿用现有模型/架构判断。代码侧只做两类小补丁：

1. 白名单补齐或排除差异场景，使用现有 matcher 生效。
2. `vllm_adapter.py` 局部读取新增 sparse/spec 字段：`strategy=fp8` 只注入 `kv_cache_dtype=fp8`，不再注入 `calculate_kv_scales`；GLM-4.7 Ascend 行读取 `speculative_token_range`。

按本次差异和 H20 卡型继承规则，需要补齐的白名单能力：

| 特性 | 模型/硬件 | 静态策略 |
|---|---|---|
| sparse | Qwen3.6-27B + L20，并按产品继承规则覆盖 H20-96/H20-141 | KVCache FP8，参数按 Q 列精确声明 |
| sparse | Qwen3.6-35B-A3B + L20，并按产品继承规则覆盖 H20-96/H20-141 | KVCache FP8，参数按 Q 列精确声明 |
| sparse | Qwen3-Embedding-0.6B + L20，并按产品继承规则覆盖 H20-96/H20-141 | KVCache FP8，参数按 Q 列精确声明 |
| sparse | DeepSeek-V4-Flash-w8a8-mtp + 910B/910C | IndexCache topk=8 |
| sparse | DeepSeek-V4-Flash + H20-96/H20-141 | IndexCache topk=4；Q 列 FP8 作为基线参数，不进入 sparse 策略 |
| sparse | zai-org/GLM-4.7 + H20-96/H20-141 | KVCache FP8，不输出源 Q 列没有声明的额外参数 |
| spec | Qwen3.5-397B-A17B-w8a8-mtp + 910B | MTP token=3，使用准确子型号行 |
| spec | Qwen3.5-397B-A17B-w8a8-mtp + 910C | MTP token=1，替换当前父 token 的隐式命中 |
| spec | DeepSeek-V4-Flash + H20-96 | MTP token=1，与 H20-141 对齐 |
| offload | DeepSeek-V4-Flash + H20-96/H20-141 | Native backend |

GLM-5.1、GLM-5 的 H20 三特性当前基础行已具备能力，修复重点是使用准确 `zai-org/GLM-*` 身份、清除旧 FP8 专用行并收紧版本排除；不能为错误展示名再增加兼容别名。

910B MemCache 行 13、14、17、Pro5000 LMCache 行 26 仍按分析口径排除，不补白名单。GLM5.1/GLM5.2 + Ascend 仍不纳入本轮功能补齐。

## 8. 同步范围

- `smart_feature_whitelist.json`：补齐差异行、删除旧 GLM FP8 专用行、收紧 GLM-5 父 token 排除、保留 910B MemCache 排除。
