# AI Infra 0 Day 白名单差异场景分析

分析日期：2026-07-16

源数据：`AI Infra工作台_0 Day测试结果_收编视图.xlsx`

配套抽取表：`AI Infra工作台_0 Day测试结果_白名单差异场景及优化参数.xlsx`

## 1. 分析口径

- 模型名称取源表 D 列，特性取 M 列，芯片取 C 列，优化参数取 Q 列。
- spec 包含 MTP、Eagle、suffix、DFlash。
- sparse 包含 IndexCache、KVCache FP8；表格中的 FP8 统一归为 sparse。
- offload 区分具体后端：Native、LMCache、MemCache、MoonCake Store。
- PD 分离不纳入本次差异判断。
- 暂不分析 GLM5.1/GLM5.2 与 Ascend 的组合场景，即源表行 12、33、45、49。
- 910B 场景不要求 MemCache，删除源表行 13、14、17 的 offload 需求。
- 源表行 26 的 Qwen-AgentWorld + Pro5000 不要求 LMCache，仅保留 suffix/spec。
- H20 同时检查 `h20-96` 和 `h20-141`，不能只以其中一种卡型代表整个 H20 场景。
- H20 三个 GLM 场景中，D 列的 `FP8` 是精度/场景标签，不属于开源模型身份。实际模型名以 Q 列启动命令为准，分别为 `zai-org/GLM-4.7`、`zai-org/GLM-5.1`、`zai-org/GLM-5`。
- 白名单按 engine 精确匹配、`model_name + model_path` 子串匹配、card token 匹配，并应用 `exclude_name_tokens` 负向过滤。

## 2. 汇总结论

本轮共纳入 38 个有效场景：

- 27 个场景未发现差异。
- 11 个场景存在功能差异或模型身份问题。
- 差异名单共 14 项：12 项确定差异、1 项 H20 型号条件差异、1 项当前功能可命中但模型身份不规范的结构风险。
- H20 + GLM 三个场景均已纳入：源表行 28 `GLM-4.7-FP8`、行 29 `GLM-5.1-FP8`、行 30 `GLM-5-FP8`。
- GLM-4.7 按正确基础名匹配后，MTP token=1 和 Native offload 已覆盖两种 H20，真实功能缺口只剩 KVCache FP8 sparse 行。
- GLM-5.1 按正确基础名匹配后，MTP token=3、IndexCache topk=4、Native offload 已覆盖两种 H20；当前问题来自旧 FP8 名称、负向排除和旧专用行之间的冲突。
- GLM-5 当前三特性均可命中，但 `GLM-5-FP8` 仍是错误模型身份，当前依赖宽子串匹配掩盖了该问题。
- 除三条 H20 GLM 外，行 47 `Kimi-K2.7-Code + 910C` 是另一条确定的模型身份不一致场景。

问题场景为源表行 8、9、10、11、15、27、28、29、30、32、47。

## 4. H20 + GLM 专项分析

| 源表行 | D 列展示名 | 实际开源模型身份 | 当前白名单表现 | 正确目标 |
|---|---|---|---|---|
| 28 | `GLM-4.7-FP8` | `zai-org/GLM-4.7` | 使用展示名时：H20-96 三特性全 miss；H20-141 命中旧 23.6.0 行，得到 MTP3、sparse、LMCache。使用正确基础名时：两种 H20 均得到 MTP1、Native，但无 sparse | MTP token=1、KVCache FP8、Native offload，两种 H20 均覆盖 |
| 29 | `GLM-5.1-FP8` | `zai-org/GLM-5.1` | 使用展示名时：H20-96 三特性全 miss；H20-141 只命中旧 sparse 行。使用正确基础名时三特性均正确 | MTP token=3、IndexCache topk=4、Native offload，两种 H20 均覆盖 |
| 30 | `GLM-5-FP8` | `zai-org/GLM-5` | 展示名和正确基础名当前都能命中基础行，因为通用 token `glm-5` 未排除 FP8；功能正确但身份不规范 | MTP token=3、IndexCache topk=4、Native offload，两种 H20 均覆盖 |

根因不是 matcher 无法处理正确模型名，而是当前白名单同时保留了两套互相竞争的策略：

1. `vllm-0.23` 基础名规则使用 `zai-org/GLM-*`，覆盖 `h20-96` 和 `h20-141`。
2. 旧 `23.6.0` 规则使用 `ZhipuAI/GLM-*-FP8`，只覆盖 H20-141，且 GLM-4.7 的 MTP token 和 offload backend 与当前 Day0 要求不同。
3. GLM-4.7、GLM-5.1 基础名规则又通过 `exclude_name_tokens=["fp8"]` 排除错误展示名，因此即使 `model_path` 已是正确基础路径，只要 `model_name` 仍含 FP8，基础行仍不会命中。

## 5. 其他类似情况检查

### 5.1 确定的同类问题

源表行 47 要求 MemCache。白名单已有 `vllm_ascend + 910c + memcache` 行，但 name token 是 `kimi-k2.7-code-w4a8`；源表 D 列是 `Kimi-K2.7-Code`，P/Q 启动路径也不带 `-w4a8`。匹配器不会自动补量化后缀，因此该行无法命中。

### 5.2 已排除范围内的潜在问题

源表行 49 的 D 列是 `GLM-5.2-w4a8`，P 列实际启动路径和白名单均是 `GLM-5.2-w8a8`。这是源表内部模型身份不一致，但属于本轮明确排除的 GLM5.2 + Ascend 场景，不计入当前差异统计。

### 5.3 可由实际路径补偿，不构成当前漏匹配

- 行 11、32：D 列是 `DeepSeek-V4-Flash-w8a8`，实际路径带 `-mtp`，可命中白名单的 `deepseek-v4-flash-w8a8-mtp`。
- 行 21：D 列是 `MiniMax-M3`，Q 列实际路径是 `MiniMax-M3-MXFP8`，可命中对应白名单行。
- 行 43：D 列是 `DeepSeek-V4-Pro-w4a8`，实际路径带 `-mtp`，可命中对应白名单行。

除上述情况外，没有再发现“白名单已有对应能力，但 D、P、Q 中实际模型身份均无法命中”的新场景。

## 7. 白名单之外的同步范围

修正白名单时还应同步以下内容：

- `tests/test_smart_feature_whitelist.py`：删除或改写旧 `ZhipuAI/GLM-*-FP8` 命中结果，补齐三个 GLM 基础名在两种 H20 上的回归断言。
- `tests/test_kv_offload_gating.py`：不再用 GLM-4.7-FP8 表示有效 LMCache 场景；需要测试 LMCache 通用逻辑时改用真实 LMCache 模型。
- `dry_run.py`：将 GLM-4.7 H20 场景改为正确基础模型名和路径。
- 源场景表或其生成规则：模型身份使用 `zai-org/GLM-*`，FP8 留在精度/特性字段，避免再次生成错误白名单。

预计不需要修改以下运行代码和配置：

- smart-feature 核心 matcher：现有 engine、模型名/路径、卡型和负向排除机制可以承载正确规则。
- spec、sparse、offload adapter：GLM-4.7 增加正确基础名 sparse 准入后，现有非 IndexCache 架构分支可生成 KVCache FP8。
- `config/defaults/nvidia_default.json`：已经使用三个正确基础名并覆盖 `h20-96`、`h20-141`。
- `reason_parser.yaml` 和 `model_utils.py` 中的旧 FP8 别名不会阻塞本次修复，可作为后续一致性清理项。
- `23.6.0_feature_matrix_summary.md` 属于历史记录，不应按当前 Day0 模型名直接改写；应在当前白名单中隔离旧策略。
