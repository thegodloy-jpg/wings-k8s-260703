# AI Infra 0 Day 白名单差异场景分析

分析日期：2026-07-16

配套抽取表：`AI Infra工作台_0 Day测试结果_白名单差异场景及优化参数.xlsx`

## 1. 分析口径

- 模型名称取源表 D 列，特性取 M 列，芯片取 C 列，优化参数取 Q 列。
- spec 包含 MTP、Eagle、suffix、DFlash。
- sparse 包含 IndexCache、KVCache FP8；表格中的 FP8 统一归为 sparse。
- offload 区分具体后端：Native、LMCache、MemCache、MoonCake Store。
- PD 分离不纳入本次差异判断。
- 暂不分析 GLM5.1/GLM5.2 与 Ascend 的组合场景，即源表行 12、33、45、49。
- 910B 场景不要求 MemCache，删除源表行 13、14、17 的 offload 需求。
- 源表行 26 的 Qwen-AgentWorld + Pro5000 不要求 LMCache，仅保留 suffix/spec。
- H20 同时检查 `h20-96` 和 `h20-141`，不能只以其中一种卡型代表整个 H20 场景。
- H20 三个 GLM 场景的模型身份严格采用 D 列：`GLM-4.7-FP8`、`GLM-5.1-FP8`、`GLM-5-FP8`。Q 列中的 `zai-org/GLM-*` 是实际权重路径，不能替代上层下发给 defaults 和白名单 matcher 的模型名。
- DeepSeek-V4-Flash 的本轮 NVIDIA 差异只来自源表行 27 的 H20 场景；`rtxpro5000-72` 是 23.6.0 历史白名单场景，不在本轮差异 Excel 中，因此不能借本轮补丁新增或增强 Pro5000-72 sparse 策略。
- 白名单按 engine 精确匹配、`model_name + model_path` 子串匹配、card token 匹配，并应用 `exclude_name_tokens` 负向过滤。

## 2. 汇总结论

本轮共纳入 38 个有效场景：

- 31 个场景未发现差异。
- 7 个场景存在功能差异或模型身份问题。
- 差异名单共 10 项：9 项确定差异、1 项 H20 型号条件差异。
- H20 + GLM 三个场景均已纳入：源表行 28 `GLM-4.7-FP8`、行 29 `GLM-5.1-FP8`、行 30 `GLM-5-FP8`。
- GLM-4.7-FP8 缺少精确 defaults 项和三特性白名单入口；其中 sparse 必须使用 `strategy=fp8`，MTP token=1，offload 使用 Native。
- GLM-5.1-FP8、GLM-5-FP8 同样缺少精确 defaults 项和白名单入口；两者 sparse 使用 IndexCache topk=4，MTP token=3，offload 使用 Native。
- 三张特性表只保留三个 FP8 精确模型行；同一轮错误适配新增的 H20 基础模型行全部删除，Q 列基础权重路径不能单独授予 smart feature。
- 行 47 `Kimi-K2.7-Code + 910C` 已按文档名称对齐，白名单中的小写 `kimi-k2.7-code` 可命中 D/P/Q 中的 `Kimi-K2.7-Code`。

问题场景为源表行 8、9、10、27、28、29、30。行 11、15、32、47 已按当前模型名口径对齐，不再作为白名单缺口。

## 4. H20 + GLM 专项分析

| 源表行 | D 列模型名 | Q 列权重路径 | 修复前表现 | 正确目标 |
|---|---|---|---|---|
| 28 | `GLM-4.7-FP8` | `zai-org/GLM-4.7` | defaults 无精确键，基础白名单排除 `fp8`，两种 H20 三特性均 miss | MTP token=1、KVCache FP8、Native offload，两种 H20 均覆盖 |
| 29 | `GLM-5.1-FP8` | `zai-org/GLM-5.1` | defaults 无精确键，基础白名单排除 `fp8`，两种 H20 三特性均 miss | MTP token=3、IndexCache topk=4、Native offload，两种 H20 均覆盖 |
| 30 | `GLM-5-FP8` | `zai-org/GLM-5` | defaults 无精确键，基础白名单排除 `fp8`，两种 H20 三特性均 miss | MTP token=3、IndexCache topk=4、Native offload，两种 H20 均覆盖 |

根因是上一轮错误地把 D 列 FP8 后缀当成展示标签，并改用 Q 列基础权重路径作为模型身份：

1. `nvidia_default.json` 只建立了基础模型项，没有三个 D 列 FP8 精确模型项。
2. `smart_feature_whitelist.json` 只保留基础名规则，并通过 `exclude_name_tokens=["fp8"]` 主动排除 D 列名称。
3. matcher 同时检查 `model_name + model_path`；即使 Q 列路径是基础名，只要上层 `model_name` 含 FP8，基础行仍会因负向过滤而拒绝命中。
4. 正确修复是在 defaults 中增加 FP8 精确项，并用三类 FP8 精确白名单行替换旧基础名行，不修改 matcher。
5. `nvidia_default.json` 中的基础模型项只提供 Q 列权重路径可能复用的普通启动参数，不授予 spec/sparse/offload；本轮删除的是三张 smart feature 表中的无来源基础行。

## 5. 其他类似情况检查

### 5.1 Kimi 名称配置错误

源表行 47 要求 MemCache，D 列是 `Kimi-K2.7-Code`，P/Q 启动路径也均为 `Kimi-K2.7-Code`。当前仓库白名单、Ascend defaults、模型族 helper、reasoning parser 和测试已统一到 `Kimi-K2.7-Code`。

白名单通过小写 token `kimi-k2.7-code` 匹配，matcher 会统一转小写，因此 `Kimi-K2.7-Code` 与 `kimi-k2.7-code` 的大小写差异不影响命中。旧 `-w4a8` 拼写通过 `exclude_name_tokens` 明确排除，模型族 helper 同样不把旧名称作为别名；reasoning parser 仅保留精确注册项。不修改通用 matcher，也不让一个父 token 同时代表两个身份。

### 5.2 已排除范围内的潜在问题

源表行 49 的 D 列是 `GLM-5.2-w4a8`，P 列实际启动路径和白名单均是 `GLM-5.2-w8a8`。这是源表内部模型身份不一致，但属于本轮明确排除的 GLM5.2 + Ascend 场景，不计入当前差异统计。

### 5.3 DeepSeek Pro5000-72 不是本轮新增依据

白名单历史上已有 `DeepSeek-V4-Flash + rtxpro5000-72` 的 spec/sparse 覆盖，其中 sparse 行来自 23.6.0 历史矩阵，不是本轮 Excel 差异。源表本轮出现的 DeepSeek-V4-Flash NVIDIA 差异是行 27 `G8600+H20 * 8`，应只补齐 `h20-96` 和 `h20-141` 的 spec/sparse/offload 缺口。

因此，当前实现应避免把 H20 行的 IndexCache 结论外推到 Pro5000-72；如果 Pro5000-72 以后需要调整，必须有对应源表行、启动参数或独立验证记录支撑。

### 5.4 可由实际路径补偿，不构成当前漏匹配

- 行 11、32：D 列短名 `DeepSeek-V4-Flash-w8a8` 不作为最终模型身份，文档核对时应改用白名单/实际路径中的 `Eco-Tech/DeepSeek-V4-Flash-w8a8-mtp`。
- 行 21：D 列短名 `MiniMax-M3` 不作为最终模型身份，文档核对时应改用白名单/实际路径中的 `MiniMax/MiniMax-M3-MXFP8`。
- 行 43：D 列按当前项目模型名使用 `DeepSeek-V4-Pro-w4a8-mtp`，可命中对应白名单行。

### 5.5 其他宽匹配结构风险

- `Qwen3.5-397B-A17B-w8a8-mtp + 910B/910C` 已按文档 D 列建立准确 `-w8a8-mtp` 行，父 token `qwen3.5-397b-a17b` 排除该后缀，避免不同卡型 token 数被宽匹配掩盖。
- H20 的 `GLM-5`、`GLM-4.7`、`GLM-5.1` 基础 token 不再保留 smart feature 行；三个 FP8 型号只使用与 D 列完全一致的独立精确行。
- 旧 `ZhipuAI/GLM-4.7-FP8`、`ZhipuAI/GLM-5.1-FP8` 的 token、卡型和后端与本轮要求不同，不能作为当前三个 FP8 精确场景的兼容行。

除上述情况外，没有再发现新的“白名单已有对应能力，但 D、P、Q 中实际模型身份均无法命中”的场景。

## 7. 最小补丁实施口径

本轮实现不重构 `config_loader`、`wings_entry`、MemCache 或 offload 后端选择链路；这些路径沿用现有统一接口。仅做以下最小补丁：

1. defaults 和白名单严格按 D 列补齐三个 H20 `GLM-*-FP8` 精确模型项。
2. Kimi 白名单、Ascend defaults、模型族 helper、reasoning parser、当前 MemCache 功能说明和测试统一改为 `Kimi-K2.7-Code`。
3. `vllm_adapter.py` 继续沿用已有局部能力：`strategy=fp8` 只注入 `kv_cache_dtype=fp8`，不注入 `calculate_kv_scales`；GLM-4.7 Ascend 行读取 `speculative_token_range`。

按本次差异和 H20 卡型继承规则，需要补齐的白名单能力：

| 特性 | 模型/硬件 | 静态策略 |
|---|---|---|
| sparse | Qwen3.6-27B + L20，并按产品继承规则覆盖 H20-96/H20-141 | KVCache FP8，参数按 Q 列精确声明 |
| sparse | Qwen3.6-35B-A3B + L20，并按产品继承规则覆盖 H20-96/H20-141 | KVCache FP8，参数按 Q 列精确声明 |
| sparse | Qwen3-Embedding-0.6B + L20，并按产品继承规则覆盖 H20-96/H20-141 | KVCache FP8，参数按 Q 列精确声明 |
| sparse | DeepSeek-V4-Flash + H20-96/H20-141 | IndexCache topk=4；Q 列 FP8 作为基线参数，不进入 sparse 策略 |
| spec/sparse/offload | GLM-4.7-FP8 + H20-96/H20-141 | MTP token=1、KVCache FP8、Native backend |
| spec/sparse/offload | GLM-5.1-FP8 + H20-96/H20-141 | MTP token=3、IndexCache topk=4、Native backend |
| spec/sparse/offload | GLM-5-FP8 + H20-96/H20-141 | MTP token=3、IndexCache topk=4、Native backend |
| spec | DeepSeek-V4-Flash + H20-96 | MTP token=1，与 H20-141 对齐 |
| offload | DeepSeek-V4-Flash + H20-96/H20-141 | Native backend |

三个 GLM FP8 精确模型项是本表 H20 GLM 场景唯一的 smart feature 入口。基础权重路径仍可参与普通 defaults 选择，但不能在缺少 FP8 D 列模型身份时获得 spec、sparse 或 offload。

910B MemCache 行 13、14、17 按最新核对口径复用 910C MemCache 卸载方式，已在白名单补齐 `backend=memcache`。Pro5000 LMCache 行 26 仍需单独核对；GLM5.1/GLM5.2 + Ascend 仍不纳入本轮功能补齐。

## 8. 同步范围

- `smart_feature_whitelist.json`：三个 GLM FP8 精确模型的三特性行替换旧基础 GLM 行，Kimi 改为文档名称，并为行 13、14、17 的 910B Qwen 场景补齐 MemCache offload。
- `nvidia_default.json`：新增 `GLM-4.7-FP8`、`GLM-5-FP8`、`GLM-5.1-FP8` 默认模型项；FP8 sparse 参数仍由白名单控制。
- `ascend_default.json`、`model_utils.py`、`reason_parser.yaml`：将 Kimi K2.7 Code 身份由错误的 `-w4a8` 改为文档中的基础名称；reasoning parser 不保留旧名称兼容项。
- `docs/features/memcache/memcache_hybrid_kimi.md`：同步当前 MemCache 契约中的 Kimi 精确名称和旧名称排除说明。
- `vllm_adapter.py`：仅保留 FP8 sparse、`speculative_token_range` 和删除 `calculate_kv_scales` 的局部补丁。
- `tests`：使用与本轮文档一致的 Kimi 名称，并增加 GLM FP8 精确命中正例和旧 Kimi 名称反例；读取旧版工作簿的 `verify_s_empty_day0_dry_run.py` 不纳入本轮修改。

验收重点：三个 D 列 GLM FP8 名称均能在 H20-96/H20-141 命中目标三特性；Kimi-K2.7-Code + 910C 命中 MemCache，旧 `-w4a8` 不命中；Qwen FP8 sparse 仍由白名单控制；行 13、14、17 的 910B Qwen 场景命中 MemCache；最终命令不出现 `calculate_kv_scales`。
