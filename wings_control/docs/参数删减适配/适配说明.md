# 参数删减适配说明

## 1. 背景

MaaS 普通参数页删除部分调优字段后，页面不再透传这些参数。需要避免两个问题：

1. 页面不传字段后，启动命令仍被 `argparse` 默认值回填，导致实际启动参数和页面语义不一致。
2. 对部分模型而言，被删除字段并不是可选项，例如 MoE 模型需要专家并行，Ascend 量化模型需要 `quantization=ascend`，DeepSeek-V4-Flash 对 `block_size` 有硬要求。

因此适配目标不是简单删除参数支持，而是把参数来源调整为：

```text
用户显式 CLI/ENV > 用户 config-file > 模型 defaults > 官方 recipe/必填兜底 > vLLM 自身默认
```

## 2. 本次删除字段

普通参数页删除的字段：

```text
dtype
kv-cache-dtype
gpu-memory-utilization
block-size
quantization
seed
quantization-param-path
enable-chunked-prefill
enable-prefix-caching
enable-expert-parallel
```

其中风险最高的是：

- `enable-expert-parallel`：页面删除后如果走 `False` 默认值，MoE 模型会静默关闭 EP。
- `gpu-memory-utilization`：页面删除后如果统一回填 `0.9`，会覆盖不同模型/卡型的官方推荐。
- `block-size`：页面删除后如果统一回填 `16`，会破坏 DeepSeek-V4-Flash 等有硬约束的模型。
- `enable-prefix-caching` / `enable-chunked-prefill`：不同模型和不同拓扑有开关差异，不能统一按 `False` 处理。

## 3. 适配原则

### 3.1 有官方 recipe 的模型直接复用

Ascend 场景优先参考 vLLM-Ascend 官方模型文档：

- DeepSeek-V4-Flash 官方 Ascend 启动命令包含 `gpu-memory-utilization`、`enable-expert-parallel`、`quantization ascend`、`block-size 128`、`enable-chunked-prefill`、`enable-prefix-caching`。
- GLM-5.2 官方 Ascend 启动命令包含 `enable-expert-parallel`、`seed 1024`、`gpu-memory-utilization 0.95`、`quantization ascend`。
- Kimi-K2.5 官方 Ascend 启动命令包含 `quantization ascend`、`no-enable-prefix-caching`、`seed 1024`、`enable-expert-parallel`、`gpu-memory-utilization 0.9`。

NVIDIA 场景优先参考 `recipes.vllm.ai`：

- DeepSeek-V4-Flash NVIDIA/DGX Station recipe 包含 `kv-cache-dtype fp8`、`block-size 256`、`gpu-memory-utilization 0.92`。
- DeepSeek-V3 / DeepSeek-R1 NVIDIA recipe 对 MoE 部署显式使用 `enable-expert-parallel`。
- Qwen3.5 小 dense 模型 recipe 没有这些删除字段，说明不应为了删参而补默认调优值。

### 3.2 没有官方 recipe 的模型只补必填字段

没有官方 recipe 或没有覆盖当前硬件/模型变体时，按以下规则判断：

| 字段 | 何时必须设置 | 不满足条件时 |
| --- | --- | --- |
| `enable_expert_parallel` | 模型架构是 MoE，且目标 engine 支持 EP | 不补，交给 vLLM 默认 |
| `quantization` | Ascend 量化权重明确需要 `ascend`，或 NVIDIA recipe 明确需要某量化算法 | 不补；不要把 `ascend` 写入 NVIDIA |
| `block_size` | 官方 recipe 或本地 adapter 证明有硬约束 | 不补，避免 `16` 误下发 |
| `gpu_memory_utilization` | 官方 recipe 给出模型/卡型推荐值 | 不补，交给 vLLM 自身默认 |
| `kv_cache_dtype` | recipe 明确要求，如 FP8/NVFP4 场景 | 不补 |
| `dtype` | recipe 明确要求，如个别 VL 模型 `bfloat16` | 不补 |
| `seed` | recipe 明确要求，或测试复现必须固定 | 不补 |
| `enable_prefix_caching` | recipe 明确开启 | 不补 |
| `no_enable_prefix_caching` | recipe 明确 `--no-enable-prefix-caching` | 不补 |
| `enable_chunked_prefill` | recipe 明确开启 | 不补 |
| `quantization_param_path` | recipe 明确给出量化参数文件 | 不补 |

## 4. defaults 如何完善

### 4.1 Ascend defaults

文件：

```text
wings_control/config/defaults/ascend_default.json
```

操作方式：

1. 在 `model_deploy_config.llm.<架构>.<模型名或 default>.vllm_ascend` 中补官方字段。
2. 同步补 `vllm_ascend_distributed`。如果官方单机和多机值不同，分别写，不要盲目复制。
3. 只补 vLLM-Ascend 生效字段，不把 MindIE 参数混入 vLLM-Ascend 模板。
4. `--no-enable-prefix-caching` 按现有配置风格写为：

```json
"no_enable_prefix_caching": true
```

5. 不要把 `max_num_seqs`、`max_num_batched_tokens` 当成本次删除字段一起处理。它们属于保留字段，并且有“页面不传 = auto”的 C9 语义，若 defaults 写死会破坏 auto。

示例方向：

```json
"DeepSeek-V4-Flash": {
  "vllm_ascend": {
    "gpu_memory_utilization": 0.9,
    "enable_expert_parallel": true,
    "quantization": "ascend",
    "block_size": 128,
    "enable_prefix_caching": true,
    "enable_chunked_prefill": true
  }
}
```

实际落值以目标版本官方 recipe 和当前已存在的 adapter 逻辑为准。

### 4.2 NVIDIA defaults

文件：

```text
wings_control/config/defaults/nvidia_default.json
```

操作方式：

1. 只参考 NVIDIA/Hopper/Blackwell recipe，不复用 Ascend recipe。
2. MoE 架构若 recipe 明确开启 EP，则补 `enable_expert_parallel: true`。
3. DeepSeek-V4-Flash 这类有 NVIDIA 硬约束的模型可补 `block_size: 256`、`kv_cache_dtype: "fp8"`、`gpu_memory_utilization` 等 recipe 字段。
4. 小 dense 模型 recipe 没有删除字段时，不补 `gpu_memory_utilization`、`block_size`、`seed` 等泛化默认值。
5. 不写 `quantization: "ascend"`。

示例方向：

```json
"DeepSeek-V4-Flash": {
  "vllm": {
    "kv_cache_dtype": "fp8",
    "block_size": 256,
    "gpu_memory_utilization": 0.92
  }
}
```

如果本地 `vllm_adapter.py` 已经对某模型强制注入硬约束，defaults 可作为显式可见的模型模板，adapter 继续保留为兼容兜底。

#### 4.5.3 JSON 修改后的最小验收

修改两个 JSON 后，至少检查以下几点：

1. `nvidia_default.json` 中不能出现 `quantization: "ascend"`。
2. `Qwen3_5ForConditionalGeneration.default` 不能再有 `enable_prefix_caching`。
3. `Qwen3_5MoeForConditionalGeneration.default` 只能保留 MoE 必填的 `enable_expert_parallel=true`，不能再有 generic `quantization`、`gpu_memory_utilization`、`enable_prefix_caching`；`Qwen3.6-35B-A3B` 专属条目必须有 `seed=1024`。
4. `DeepSeek-V4-Flash-A2/A3` 的 Ascend 值必须继续与 `tokenbox.md` 当前命令一致。
5. 小 dense、embedding、rerank 不应因为页面删参出现 argparse 默认回填的 `gpu_memory_utilization=0.9`、`block_size=16`、`seed=0`。

### 4.6 修改顺序建议

按风险从高到低处理：

1. 先改 `config_loader._set_common_params()`，阻止删除字段在非显式情况下由 argparse 默认值回填。
2. 再改 `ascend_default.json` 中已有官方 recipe 的条目：DeepSeek-V4、GLM-5.2、Qwen3.6、Kimi-K2.5。
3. 再改 `nvidia_default.json` 中 DeepSeek-V4-Flash 和 MoE EP 必填项。
4. 最后跑 dry-run，确认小 dense、embedding、rerank 不再收到删除字段默认值。

## 5. 代码层需要调整

仅补 defaults 不能完全解决问题。当前参数链路是：

```text
wings_start.sh / ENV / CLI
  -> start_args_compat.py argparse
  -> LaunchArgs
  -> config_loader.load_and_merge_configs()
  -> engine_config
  -> vllm_adapter.py
  -> start_command.sh
```

`start_args_compat.py` 当前仍会给删除字段设置默认值，例如：

```text
dtype=auto
kv_cache_dtype=auto
gpu_memory_utilization=0.9
block_size=16
seed=0
enable_chunked_prefill=false
enable_prefix_caching=false
enable_expert_parallel=false
```

`config_loader._set_common_params()` 当前逻辑是：

```text
用户显式传参 -> 覆盖模板
用户未显式传参，模板已有值 -> 保留模板
用户未显式传参，模板没有值 -> 用 argparse 默认值补入
```

这会导致页面删除字段后，模板没有值的模型仍然收到 argparse 默认值。需要增加“删除字段非显式不回填”规则：

```text
如果 key 属于页面删除字段
且 key 不是 CLI/ENV 显式传入
且 defaults/config-file 没有设置该字段
则不要把 argparse 默认值写入 engine_config
```

建议实现位置：

```text
wings_control/core/config_loader.py::_set_common_params()
```

可新增集合：

```python
REMOVED_PAGE_TUNING_KEYS = {
    "dtype",
    "kv_cache_dtype",
    "quantization",
    "quantization_param_path",
    "gpu_memory_utilization",
    "enable_chunked_prefill",
    "block_size",
    "seed",
    "enable_expert_parallel",
    "enable_prefix_caching",
}
```

然后在 `elif value not in params:` 前判断：

```text
key in REMOVED_PAGE_TUNING_KEYS 且 key not in explicit_keys -> continue
```

注意：如果 defaults 已经有值，应继续保留 defaults；如果用户显式传 CLI/ENV，应继续覆盖 defaults。

## 6. 适配流程

每新增或修正一个模型，按下面顺序处理：

1. 确认硬件场景：Ascend / NVIDIA。
2. 确认 engine：`vllm_ascend` / `vllm_ascend_distributed` / `vllm` / `vllm_distributed`。
3. 确认模型架构和模型变体：dense、MoE、VL、embedding、reranker、W8A8、W4A8、FP8、NVFP4。
4. 查官方 recipe：
   - Ascend 查 vLLM-Ascend model tutorials。
   - NVIDIA 查 `recipes.vllm.ai` 对应模型页。
5. 只摘取本次删除字段，以及与这些字段强绑定的正/反向开关。
6. 写入对应 defaults 模板。
7. 没有官方 recipe 时，只按“必填字段判断表”补值。
8. 检查 `_set_common_params()` 是否会把删除字段的 argparse 默认值重新补回。
9. dry-run 生成启动命令，确认字段来源符合预期。

## 7. 验证建议

### 7.1 静态验证

检查 JSON：

```powershell
python -m json.tool wings_control/config/defaults/ascend_default.json > $null
python -m json.tool wings_control/config/defaults/nvidia_default.json > $null
```

检查删除字段是否只出现在有依据的模板：

```powershell
rg -n "gpu_memory_utilization|block_size|quantization|seed|kv_cache_dtype|enable_chunked_prefill|enable_prefix_caching|enable_expert_parallel|no_enable_prefix_caching" wings_control/config/defaults
```

### 7.2 dry-run 验证

至少覆盖三类样例：

1. 有官方 Ascend recipe 的 MoE 量化模型：确认官方字段进入最终命令。
2. 有 NVIDIA recipe 的 MoE/FP8/FP4 模型：确认 NVIDIA 专属字段进入最终命令，且没有 `quantization=ascend`。
3. 没有删除字段要求的小 dense 模型：确认命令里不再出现 argparse 默认回填的删除字段。

### 7.3 回归关注点

- 用户显式 CLI/ENV 仍应优先覆盖 defaults。
- config-file 中显式写入的字段应优先于模型 defaults。
- `max_num_seqs` / `max_num_batched_tokens` 的 C9 auto 语义不能被本次修改破坏。
- embedding / reranker 仍不能误开 prefix caching 或 chunked prefill。
- PD 配置和模型 defaults 不要互相污染；PD 专属值继续放在 `pd_config.json`。
