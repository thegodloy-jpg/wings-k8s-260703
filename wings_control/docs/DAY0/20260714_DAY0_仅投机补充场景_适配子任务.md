# 补充子任务：仅投机三场景适配

## 任务目标与范围

本子任务覆盖母文档中原先从 16 个 S 列为空主清单中剔除、但现在需要单独适配的 3 个“仅投机”场景。目标是把这 3 个场景的 `--speculative-config` 适配到当前项目的 smart feature、命令生成和验收链路中，并补齐进入对应场景所需的最小普通启动 default/profile。

- 母文档：[20260713_220329_S列为空场景四类整理.md](./20260713_220329_S列为空场景四类整理.md)
- 补充场景：`Qwen-AgentWorld-35B-A3B-BF16`、`MiniMax-M2.7-W8A8 910C`、`MiniMax-M2.7-W8A8 910B`
- 本子任务只处理投机解码及其对应的普通启动 profile，不处理 Mooncake、LMCache、MemCache、native offload、sparse、IndexCache 或其它卸载能力。

本文件是执行指导，不表示适配已经完成。完整命令证据仍以母文档和后续沉淀的标准命令 / scenario JSON 为准。

## 事实源与判断规则

- 适配目标不是输出与样例命令逐字一致，而是最终 `--speculative-config` 语义正确。
- `enable_speculative_decode=false` 时，三个场景都不得输出 `--speculative-config`。
- `enable_speculative_decode=true` 时，才允许进入自动投机合成。
- 若上层已经显式提供 `engine_config.speculative_config`，当前项目不应再自动合成第二份 speculative config。
- 真实 draft path 优先；`none`、`null`、空字符串和等价哨兵值都应视为无 draft。
- 依赖草稿模型的场景不能伪造 draft。无真实 draft path 时，不得输出 `eagle3` / `draft_model`，应按 suffix 兜底。
- 本子任务的核心验收仍看 `--speculative-config`。普通 default/profile 需要补齐到能稳定进入场景，但 TP / DP、host / port、served model name、parser、prefix caching、普通 max 参数与样例命令的逐字差异不作为投机通过条件。

## 期望 speculative config

### Qwen-AgentWorld

开关开启且模型 / 卡型命中时，目标为：

```json
{"method":"suffix","num_speculative_tokens":32}
```

注意事项：

- 该场景应通过 exact spec 能力命中或等价策略生成 suffix 32。
- 不应落到当前通用 suffix floor 的 5 tokens。
- 不需要 draft model path。
- 不应因为 Qwen / MoE 家族其它场景支持 MTP 就改成 `method=mtp`。

### MiniMax-M2.7-W8A8 / QuaRot

用户提供真实 Eagle3 draft path 时，目标为：

```json
{
  "enforce_eager": true,
  "method": "eagle3",
  "model": "<真实 Eagle3 draft path>",
  "num_speculative_tokens": 3
}
```

注意事项：

- `model` 必须来自真实 `speculative_decode_model_path`。
- 无真实 draft path 时不能输出 `method=eagle3`。
- 不要只在白名单里写 `mtp_method=eagle3`。这样容易绕过 draft 分支，导致最终 config 缺少 `model` 字段。
- `enforce_eager=true` 是 speculative config 内部字段；不要把它和顶层 `--enforce-eager` 混为同一个验收项。

无 draft 时，目标行为为：

```json
{"method":"suffix", "...":"按当前项目 suffix 兜底规则生成"}
```

suffix 兜底的具体 token 数如果没有新证据，沿用当前项目规则；验收重点是“不能输出 Eagle3”。

## 实施顺序

1. C0：沉淀 3 个 scenario JSON，明确 engine、model name、model path、card token、投机开关和 draft path 输入。
2. C1：先做 Qwen-AgentWorld suffix 32。该场景不依赖 draft，能快速验证 exact spec 命中和 suffix token 覆盖。
3. C2：做 MiniMax-M2.7-W8A8 910C / 910B 的 exact 模型识别和 spec 能力命中。
4. C3：补 MiniMax Eagle3 draft 分支，确保真实 draft path 时输出 Eagle3 3 tokens + `enforce_eager=true`。
5. C4：补负向校验：开关关闭、draft 为 `none/null/空`、模型不命中时，不输出错误的 Eagle3 或多余 speculative config。

## 禁止事项

- 禁止把 Mooncake、LMCache、MemCache、native offload 加入本子任务验收范围。
- 禁止用 `mtp_method=eagle3` 直接生成 MiniMax Eagle3 config；Eagle3 必须带真实 `model` 字段。
- 禁止在没有真实 draft path 时输出 `method=eagle3` 或 `method=draft_model`。
- 禁止把 Qwen-AgentWorld suffix 32 改成 MTP。
- 禁止把通用 suffix 5 tokens 当成 Qwen-AgentWorld 的目标输出。
- 禁止无条件输出 reasoning parser、function call parser 或顶层 `--enforce-eager` 并把它们算作投机适配通过。
- 禁止用显示精度列拼接模型名；只信任真实 `model_name` / `model_path`。

## 交付物与完成定义

每个场景至少交付：

- scenario JSON 或等价标准输入。
- 投机开关关闭 / 开启的 dry-run 输出。
- 若是 MiniMax，还需要至少一组真实 Eagle3 draft path 输入和一组无 draft 输入。
- 当前项目生成的最终命令片段，重点展示 `--speculative-config`。
- `comparison.md` 或等价说明，记录通过项、允许差异和必须修复项。

完成状态只允许三类：

- `supported + semantic pass`：当前项目能按输入生成目标 speculative config。
- `supported with suffix fallback`：草稿模型缺失，项目正确回落 suffix，且没有错误输出 Eagle3。
- `blocked by external evidence`：缺少真实 Eagle3 draft path 或真实模型 ID，无法从仓库内确认。
