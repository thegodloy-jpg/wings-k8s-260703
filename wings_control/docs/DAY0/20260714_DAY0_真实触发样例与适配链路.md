# DAY0 真实触发样例与适配链路

本轮 DAY0 适配的主线是把“页面请求 + 模型/卡型识别 + 白名单能力 + default 静态配置 + adapter 动态渲染”串成一条单一生效链路：启动时先由 `config_loader._get_model_specific_config()` / `_match_model_engine_config()` 根据模型名、`config.json` 架构、硬件卡型选择对应的普通 defaults，例如 MiniMax-M2.7-w8a8-QuaRot 在 Ascend910B/910C 下命中 `ascend_default.json` 的精确 profile，Qwen-AgentWorld 命中 `nvidia_default.json` 的 Pro5000 profile，Kimi-K2.6-W4A8 命中 Kimi 0.21 profile；这些 defaults 只承载稳定静态字段，例如 quantization、load_format、parser、max_model_len、max_num_seqs、additional_config、compilation_config 等，不再把 `served_model_name`、TP/DP、端口、draft model path 这类页面输入或运行时推导字段写死。随后 `apply_effective_feature_enablement()` 用页面开关与 `smart_feature_whitelist.json` 做交集，白名单只声明能力和静态策略：Qwen-AgentWorld 在 Pro5000 下声明 suffix 32，MiniMax-M2.7-NVFP4 在 Pro5000 下声明 spec suffix 10 且 offload backend=lmcache，Kimi-K2.6-W4A8 在 910C 下声明 offload=memcache 且 DFlash 需要真实 draft path，MiniMax-M2.7-w8a8-QuaRot 在 910B/910C 下声明 `draft_method=eagle3`、token=3、`draft_enforce_eager=true`；进入 `vllm_adapter` 后，`_prepare_engine_config()` 再补运行时拓扑和安全字段，例如 Kimi-K2.6 在 16 卡上补 `TP=4/DP=4`，MiniMax-M2.7-NVFP4 Pro5000 由专属逻辑补 `TP=min(4,device_count)` 和 `DP=device_count/TP`，GLM/DeepSeek 等其它模型继续走各自既有 resolver。最终命令渲染时，`_build_speculative_cmd()` 统一读取白名单行：MTP/suffix 直接按白名单 token 输出，Eagle3/DFlash 这类 draft 策略必须同时命中白名单并提供真实 `speculative_decode_model_path`，否则不能伪造 draft，统一回退 suffix；因此 MiniMax-M2.7-w8a8-QuaRot 有真实 Eagle3 draft 时输出 `method=eagle3, model=<draft>, draft_tensor_parallel_size=1, num_speculative_tokens=3, enforce_eager=true`，没有 draft 时回 suffix，Kimi-K2.6 有真实 DFlash draft 时输出 `method=dflash, tokens=15`，没有 DFlash 时回 suffix，Qwen-AgentWorld 不依赖 draft，直接输出 suffix 32。KV offload 同样按 backend 分流：native 输出 `--kv-offloading-backend native --kv-offloading-size <KV_MEM_OFFLOAD_SIZE>`，lmcache 输出 `LMCacheConnectorV1`/LMCache env，memcache 输出 `AscendStoreConnector + memcache`；其中 Pro5000 上的 MiniMax-M2.7-NVFP4 是 LMCache connector 场景但不执行 `install.py --lmcache-target nvidia-x86`，因为 NVIDIA/vLLM 的 LMCache runtime 由镜像提供，Kimi MemCache 也不走 LMCache patch。最后，投机推理只通过 vLLM CLI 表达，不再触发 EARS/runtime-deps 安装，不再注入 `WINGS_ENGINE_PATCH_OPTIONS` 的 `ears`、`VLLM_EARS_TOLERANCE` 或 `--install-runtime-deps`；`advanced_features.json` 由 `wings_entry._write_advanced_features_json()` 写出最终有效状态，`features` 表示开关和白名单交集后的能力是否生效，`variants` 展示实际策略如 `suffix`、`mtp`、`eagle3`、`dflash`、`native_kv_offloading_backend` 等。整体适配目标不是让命令逐字等于 Excel，而是保证真实输入条件下 default 命中准确、白名单边界清晰、draft 不伪造、offload 后端互斥、EARS 不再参与、最终 CLI 与状态展示语义一致。
