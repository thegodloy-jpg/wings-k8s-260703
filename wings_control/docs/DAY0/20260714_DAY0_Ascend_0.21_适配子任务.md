# 子任务一：Ascend vLLM-Ascend 0.21 场景适配

## 任务目标与范围

本子任务覆盖母文档中原 16 个待适配场景里的 Ascend 部分，目标是把 `vllm_ascend` / `vllm-ascend-0.21` 相关场景适配到当前项目的 defaults、smart feature、命令生成和验收链路中。

- 母文档：[20260713_220329_S列为空场景四类整理.md](./20260713_220329_S列为空场景四类整理.md)
- 原 16 个场景内 Ascend 行：`11`、`18`、`30`、`44`、`45`、`41`、`17`、`40`、`42`
- 补充纳入的双机场景：GLM4.7 910B 双机、GLM4.7 910C 双机
- 参考文档：`deepseek_v4_flash_vllm_ascend_021_adaptation.md`、`kimi_k26_k27_memcache_dflash_adaptation.md`、`qwen35_qwen36_adaptation_design.md`

本文件是执行指导，不表示适配已经完成。完整命令证据仍以母文档逐行记录为准。

## 事实源与判断规则

- 命令源规则：优先使用母文档中对应行 Q 列启动命令；Q 列没有可执行启动命令时，回退到 O 列原文。
- M 列规则：逐场景保留 M 列特性加速分析。M 为空时不得从相邻模型或其它平台推断加速特性已经生效。
- 适配目标不是输出与 Excel 文本一模一样的命令。TP / DP、host / port、served model name、max params、prefix caching 等会由用户输入、CLI、ENV、defaults、adapter 和开关共同决定。验收看最终语义是否一致。
- function call、reasoning / thinking、MTP / suffix spec、sparse、offload 都必须受页面开关、白名单和模型支持能力共同控制。
- `spec`、`sparse`、`offload` 要区分“用户请求”与“实际生效”。禁止只看页面开关或 Excel 命令就认为特性已生效。
- `native`、`LMCache`、`MemCache` 是互斥路径。Ascend Kimi 场景涉及 MemCache 时，不得混入 LMCache 或 NVIDIA native offload。
- offload 容量以 `KV_MEM_OFFLOAD_SIZE` 或当前项目解析结果为准，不硬编码 Excel 样例中的 20GB / 40GB。
- DFlash 只有在真实 DFlash draft model path 存在且模型支持时才生效；没有 DFlash draft 时不得输出 `method=dflash`，但投机开关有效且 spec 白名单允许时应按通用规则回落 suffix spec。

## 实施顺序

1. A0：锁定真实模型名、模型路径、卡型描述、节点数、每节点卡数和页面开关输入，形成每个场景的 scenario JSON。
2. A1：先验证现有或接近现有的单机场景：`11`、`30`、`18`。这一步主要确认当前命令生成链路和 baseline 边界。
3. A2：处理上一版本对齐场景：`17`、`42`、`40`。重点是 exact matcher、白名单行和 topology 不能互相污染。
4. A3：处理分布式场景：`41`、GLM4.7 910B 双机、GLM4.7 910C 双机。每个双机场景都需要 node0 / node1 独立产物。
5. A4：处理 Kimi 专项：先 `45`，再 `44`。只有 DFlash draft 证据缺失时阻塞 DFlash 分支，不阻塞 MemCache / parser / baseline 适配。

## 动态参数与禁止事项

- 禁止硬编码 Excel 中的 TP / DP / global rank 样例。最终值应从用户输入、节点拓扑、defaults 和 adapter 规则派生。
- 禁止把 M 列为空的场景补成“默认有加速”。如果需要启用加速，必须有白名单、开关和模型能力证据。
- 禁止跨 backend 复制能力：Ascend MemCache、NVIDIA native offload、LMCacheConnectorV1 不能混用。
- 禁止把 DeepSeek V4 Flash、V4 Pro、DeepSeek-Coder-V2 视为同一个白名单模型。
- 禁止把 GLM4.7 910B 和 910C 的拓扑、TP / DP、双机 rank 直接互相套用。
- 禁止在没有 DFlash draft path 的 Kimi K2.6 场景里输出 DFlash speculative config；这不等于关闭 spec，需保留 suffix 兜底语义。
- 禁止无条件输出 function / reasoning parser 参数。它们必须受开关和模型支持能力控制。

## 交付物与完成定义

每个场景至少交付：

- scenario JSON 或等价标准输入。
- 母文档命令证据引用，或说明该行无可执行命令只能做 baseline。
- 当前项目生成的最终命令。
- `merged_params.json`、`advanced_features.json`、`/v1/startup/accel` 关键片段。
- `comparison.md`，写明语义一致项、允许差异项和必须修复项。

双机场景额外要求：

- node0 / node1 启动脚本。
- rank、`data_parallel_start_rank`、RPC 端口、网卡、headless 的逐节点说明。
- 两节点参数差异必须是拓扑差异，不能是随机或拷贝遗漏。

完成状态只允许三类：

- `supported + semantic pass`：当前项目已能按输入生成语义正确命令。
- `baseline`：该场景没有优化证据，只保留基础启动能力和明确关闭特性。
- `blocked by external evidence`：缺少真实模型路径、draft path、镜像能力或双机拓扑输入，且无法从仓库内确认。
