# DAY0 S列为空场景适配修改说明

## 模型命名规则

本轮明确不根据展示列的精度信息派生模型名。原则如下：

- 上层下发的 `model_name` / `model_path` 必须优先使用真实开源模型名或 Q 列命令中的真实路径。
- 只有开源模型名本身包含精度或量化后缀时才保留后缀，例如 `MiniMax-M3-MXFP8`、`MiniMax-M2.5-NVFP4`、`Qwen3.5-397B-A17B-NVFP4`。
- `Qwen3.5-122B-A10B`、`Qwen3.5-27B`、`Qwen3.5-35B-A3B` 的 Q 列命令路径不带 `BF16`，所以 defaults key、白名单 `name_tokens`、dry-run 场景名和测试输入均不再额外拼接 `BF16`。

## Ascend 0.21 source 收敛

本轮已触达的 Ascend DAY0 适配行统一使用对应引擎版本 `vllm-ascend-0.21`：

- `DeepSeek-Coder-V2-Instruct-BF16` spec 行从临时来源改为 `vllm-ascend-0.21`。
- `GLM-4.7-W8A8-floatmtp` 910B / 910C spec 行改为 `vllm-ascend-0.21`。
- `Kimi-K2.6` / `Kimi-2.6` spec 和 MemCache offload 行使用 `vllm-ascend-0.21`。
- `Kimi-K2.7-Code` MemCache offload 行使用 `vllm-ascend-0.21`，且不允许自动 speculative config。

## 关键代码改动

- `model_utils.py`
  - 增加 Kimi-K2.6 / Kimi-K2.7-Code 族识别。
  - 收敛 RTX PRO 5000 72G 判定，兼容 `G6550+RTX PRO 5000 * 8`、`TokenBox RTX PRO 5000 * 8`、`NVIDIA RTX PRO 5000 72GB` 等写法。
  - 增加 `MiniMax-M3-MXFP8 + Pro 5000 + vllm` 精确判定。

- `config_loader.py`
  - defaults 查找增加 basename 候选，使 `Qwen/Qwen3.5-27B` 能命中 `Qwen3.5-27B` exact profile。
  - Kimi-K2.7-Code 在有效特性收口处关闭 spec，避免 Kimi-K2.6 的 DFlash/suffix 能力扩散。

- `vllm_adapter.py`
  - MiniMax-M3-MXFP8 Pro 5000 注入 `VLLM_USE_V1=1`、`PYTHONHASHSEED=0`。
  - native offload CLI 统一从白名单 backend 解析，输出 `--kv-offloading-backend native --kv-offloading-size <N>`。
  - Kimi-K2.6 支持 DFlash draft path，Kimi-K2.7-Code 禁止自动 spec。
  - MiniMax-M2.7 继续走 LMCache 配方，与 MiniMax-M2.5 / M3 native 分支隔离。

- `wings_entry.py`
  - NVIDIA/vLLM 场景不再安装 `install.py --lmcache-target nvidia-x86`。
  - native、MemCache、LMCache 作为互斥 backend 处理，避免同一启动脚本混用多套卸载实现。
