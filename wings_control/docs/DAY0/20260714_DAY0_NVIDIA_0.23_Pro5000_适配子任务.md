# 子任务二：NVIDIA vLLM 0.23 Pro 5000 场景适配

## 任务目标与范围

本子任务覆盖母文档中原 16 个待适配场景里的 NVIDIA Pro 5000 部分，目标是把 `vllm` / `vllm-0.23` / RTX PRO 5000 相关场景适配到当前项目的 defaults、smart feature、命令生成和验收链路中。

- 母文档：[20260713_220329_S列为空场景四类整理.md](./20260713_220329_S列为空场景四类整理.md)
- 原 16 个场景内 NVIDIA 行：`20`、`21`、`22`、`23`、`24`、`48`、`49`
- 参考文档：`nvidia_h20_l20_day0_adaptation_analysis.md`、`qwen35_qwen36_adaptation_design.md`

本文件是执行指导，不表示适配已经完成。完整命令证据仍以母文档逐行记录为准。

## 事实源与判断规则

- 命令源规则：优先使用母文档中对应行 Q 列启动命令；Q 列没有可执行启动命令时，回退到 O 列原文。本子任务 7 行均以 Q 列命令作为首要证据。
- M 列规则：逐场景保留 M 列特性加速分析，不从模型族或相邻精度自动推断其它特性。
- 适配目标不是输出与 Excel 文本一模一样的命令。TP / DP、host / port、served model name、max params、prefix caching 等会由用户输入、CLI、ENV、defaults、adapter 和开关共同决定。验收看最终语义是否一致。
- function call、reasoning / thinking、MTP / suffix spec、sparse、offload 都必须受页面开关、白名单和模型支持能力共同控制。
- `native`、`LMCache`、`MemCache` 是互斥路径。Pro 5000 native offload 场景不得输出 LMCacheConnectorV1；MiniMax M2.7 LMCache 场景不得输出 native backend。
- offload 容量以 `KV_MEM_OFFLOAD_SIZE` 或当前项目解析结果为准，不硬编码 Excel 样例值，也不把 LMCache 的 per-card 公式套到 native offload。
- 需要统一卡型识别：白名单建议使用 `rtxpro5000-72`，defaults 硬件 key 建议使用 `rtx_pro_5000_72G`，`G6550+RTX PRO 5000 * 8` 和 `TokenBox RTX PRO 5000 * 8` 只做一次标准化。

## 实施顺序

1. B0：统一真实 `model_name`、`model_path`、卡型 token、节点数、每节点卡数和页面开关输入，形成每个场景的 scenario JSON。
2. B1：先验证已接近现有路径的 `48`、`49`。这两行能快速暴露 Pro 5000 卡型标准化、NVFP4 env、offload backend 和 MTP token 的问题。
3. B2：处理 Qwen 系列 `22`、`23`、`24`。重点是同族不同模型的 spec / native 差异，尤其是 `24` 不应因为同族支持 MTP 就自动启用 spec。
4. B3：处理 MiniMax `20`、`21`。重点是 exact model token、suffix spec 和 native offload，不让 M2.7 LMCache 逻辑污染 M2.5 / M3。

## 代码落点与责任边界

| 适配内容 | 首选落点 | 注意事项 |
| --- | --- | --- |
| NVIDIA 普通 defaults、Pro 5000 profile、NVFP4 env | `wings_control/config/defaults/nvidia_default.json` | defaults key 统一，避免在每个白名单行重复写卡型别名。 |

| function call / reasoning parser | `wings_control/docs/features/reasoning_parser/reason_parser.yaml`、`wings_control/core/config_loader.py` | parser 参数由开关、模型能力和白名单共同决定。 |
| 命令验收 | `wings_control/docs/启动命令标准化校验/startup_command_standard.py`、`wings_control/docs/DAY0/verify_qwen_day0_dry_run.py` | 使用语义比对，不做纯字符串 diff。 |

## 动态参数与禁止事项

- 禁止硬编码 Excel 中的 TP / DP / port / served model name。最终值应从 scenario JSON、用户输入、defaults 和 adapter 规则派生。
- 禁止把 `G6550+RTX PRO 5000 * 8`、`TokenBox RTX PRO 5000 * 8` 作为两套能力维护。应在卡型识别层标准化后复用。
- 禁止把 native offload 和 LMCacheConnectorV1 同时输出。行 `48` 是 LMCache，行 `20`、`21`、`22`、`23`、`24`、`49` 是 native。
- 禁止把 `KV_MEM_OFFLOAD_SIZE` 写死成样例容量。页面或 ENV 未给出时，应明确 disabled / unsupported / missing input。
- 禁止给行 `24` 自动补 speculative config。
- 禁止把 MiniMax M2.7 的 LMCache 默认逻辑套到 MiniMax M2.5 / M3。
- 禁止无条件输出 function / reasoning parser 参数。它们必须受开关和模型支持能力控制。

## 验收矩阵

| 验收项 | 必验内容 |
| --- | --- |
| 命令产物 | `start_command.sh` 或等价最终启动脚本中 engine 为 `vllm`，版本目标为 vLLM 0.23 语义。 |
| 参数产物 | `merged_params.json` 中 model、card、TP / DP、spec、offload、parser 状态可追踪。 |
| offload backend | native 行不得出现 `kv_transfer_config` / LMCache env；行 `48` 必须是 LMCacheConnectorV1 且不得出现 native backend。 |
| spec 状态 | suffix / MTP 方法、token 数和 draft / non-draft 路径符合 M 列和白名单结果。 |
| 加速状态 | `advanced_features.json` 和 `/v1/startup/accel` 能区分 requested / effective / disabled / unsupported。 |
| 负向校验 | 关闭开关、模型不命中白名单、offload size 为空时，不应输出对应有效参数。 |
| 语义比对 | 用标准化校验工具对比当前项目最终命令和母文档 Q 列证据，记录允许差异与真实偏差。 |

## 阻塞项与处理原则

- 行 `20` 至 `24` 的真实模型 ID、镜像依赖和 Pro 5000 卡型字符串需要锁定。缺一项时只阻塞对应 exact 场景，不阻塞其它行。
- MiniMax M3 的镜像安装依赖属于部署前置，不能作为 adapter 命令拼接逻辑。
- `KV_MEM_OFFLOAD_SIZE` 的来源需要明确。没有输入时不要用样例值替代。
- 卡型标准化必须先收敛，否则 defaults、白名单和校验工具会出现同场景多套 token。

## 交付物与完成定义

每个场景至少交付：

- scenario JSON 或等价标准输入。
- 母文档 Q 列命令证据引用。
- 当前项目生成的最终命令。
- `merged_params.json`、`advanced_features.json`、`/v1/startup/accel` 关键片段。
- `comparison.md`，写明语义一致项、允许差异项和必须修复项。

完成状态只允许三类：

- `supported + semantic pass`：当前项目已能按输入生成语义正确命令。
- `supported with documented delta`：当前项目输出与 Excel 文本不同，但差异来自当前动态配置或统一标准化，且语义可解释。
- `blocked by external evidence`：缺少真实模型路径、镜像能力、offload size 或卡型输入，且无法从仓库内确认。
