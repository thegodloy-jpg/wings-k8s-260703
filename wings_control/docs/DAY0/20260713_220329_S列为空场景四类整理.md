## 处理口径

1. 先从 S 列为空的 23 个场景中，剔除 GLM-5.1 + 昇腾、GLM-5.2 + 昇腾场景。
2. 将 `G6550+RTX PRO 5000 * 8` 和 `TokenBox RTX PRO 5000 * 8` 统一归为 Pro 5000。
3. 昇腾场景统一对应 `vllm-ascend-0.21`，NV / Pro 5000 场景统一对应 `vllm-0.23`。
4. 再剔除以下 3 个场景：
   - Qwen-AgentWorld-35B-A3B-BF16 + G6550+RTX PRO 5000 * 8 + 1卡
   - MiniMax-M2.7-W8A8 + G8680V3+910C*8 + 单机
   - MiniMax-M2.7-W8A8 + G5680+910B*8 + 单机
5. 在剩余 16 个场景中，将 DeepSeek-Coder-V2-Instruct-BF16、GLM4.7-W8A8-floatmtp 910B、GLM4.7-W8A8-floatmtp 910C 归入“与上一版本对齐”。
6. 将 DeepSeek V4-Pro-W4A8 + G8680V3+910C*8 独立归入双机场景。

## 当前项目适配注意事项

### 总体口径

- 本文中的启动命令来自旧版收编视图，只作为场景参数证据；落到当前项目时，以 `config_loader.load_and_merge_configs()`、`apply_effective_feature_enablement()`、`_set_kv_cache_config()`、`engine_manager.start_engine_service()` 到 `vllm_adapter.build_start_script()` 的实际生成链路为准。
- 适配目标不是保证 Wings 输出命令与 Excel / 官方样例逐字一致，而是保证最终启动语义一致且由当前项目入口正确收敛。TP / DP、host / port、served_model_name、max_model_len、enable_prefix_caching 等会受用户显式 CLI / ENV、config-file、defaults 和 adapter 特例共同影响；function call、reasoning parser / thinking、MTP / suffix、sparse / IndexCache、offload backend 则由对应开关、白名单和模型支持共同决定。验收时应比对最终语义与关键参数，不以命令文本完全相同为目标。
- `spec`、`sparse`、`offload` 都区分“请求态”和“有效态”。页面、接口、环境变量只能表达请求；最终是否生成参数由 `smart_feature_whitelist.json` 按 `engine + name_tokens + card_tokens` 收敛。Ascend 场景如果解析不出 `910b` / `910c`，对应白名单会全部 miss。
- 普通启动参数优先进入 defaults 或明确的 scenario profile；TP / DP、node rank、DP start rank、MemCache 容量等运行时派生值不要照抄 Excel 命令固化。
- `native`、`LMCache`、`MemCache` 三类 offload backend 必须互斥。MemCache 走 `wings_control/features/kv_offload/memcache/` 的 helper 和 `config_loader.py` 中的 `AscendStoreConnector` 注入；不要在 `vllm_adapter.py` 里拼 `ock.mmc.*` 文本。
- offload 容量入口统一使用页面或环境变量下发的 `KV_MEM_OFFLOAD_SIZE`。不要保留样例里的 `20GB`、`40GB` 等固定默认；LMCache / MemCache 都需要按当前项目规则换算成最终每卡容量或 backend 所需容量。
- 投机解码必须验证最终 `--speculative-config`。`enable_speculative_decode=true` 不是 MTP 已生效的证明；真实 draft path 优先走 draft / eagle 分支，无 draft 时才走白名单 MTP 或 suffix fallback。
- `--host 0.0.0.0` 是直接拉 engine 的常见写法；通过 Wings sidecar/proxy 包装时，引擎侧 host 可能按运行时策略改为 `127.0.0.1` 或其它内部地址，验收时应看最终生成的 `start_command.sh`。

### 容易遗漏的动态控制项

- 显式覆盖优先级：CLI / ENV 显式参数、config-file、model defaults、hardware defaults 会共同决定最终值。尤其 `DATA_PARALLEL_SIZE`、`TENSOR_PARALLEL_SIZE`、`MAX_MODEL_LEN`、`MAX_NUM_BATCHED_TOKENS`、`ENABLE_PREFIX_CACHING` 等不要仅按 Excel 命令反推。
- TP / DP 拓扑：PD、`dp_deployment`、GLM5.2、DeepSeek-V4-Flash、MiniMax-M2.7、GLM4.7 W8A8 等都有 adapter 或 config_loader 特例；同一场景在不同卡数、节点数、用户输入下，最终 TP / DP 可以不同。
- Function Call：只有用户打开 `enable_auto_tool_choice` 且场景/模型能解析出 `tool_call_parser` 时才应输出相关参数；否则当前项目会主动移除 `tool_call_parser` 与 `enable_auto_tool_choice`。
- Reasoning / Thinking：`enable_auto_think_choice` 控制 `reasoning_parser` 与默认 thinking 参数，但 `chat_template_kwargs` 的请求级输入仍可能覆盖默认值；Qwen / GLM 与 DeepSeek / Kimi 的字段名也不同。
- MTP / Spec：若用户或配置已显式提供 `engine_config.speculative_config`，当前项目不会再自动合成；自动 MTP 只应在 `enable_speculative_decode=true` 且白名单命中有效 `spec` 后生成。
- Sparse / IndexCache：不是看到 Excel 中 `hf_overrides`、`--enable-prefix-caching` 就直接照抄，而是由 `enable_sparse`、白名单、模型族和 adapter 共同决定最终 `hf_overrides` / `additional_config`。
- Offload backend：native offload、LMCache、MemCache 需要按后端互斥归一，并同步检查 `enable_prefix_caching`、`kv_transfer_config`、`kv_events_config`、MemCache 容量派生逻辑。
- 启动回显与验收：`/v1/startup/accel`、日志里的启动命令、最终 start script 应在 TP / DP、offload、spec、sparse 等有效态收敛后再比对，避免用预合并或请求态字段做结论。

### 阻塞项与外部确认

- Kimi K2.6 DFlash：必须确认真实 DFlash draft model path 和镜像内依赖是否存在。没有 draft path 时，行 44 只能按 MemCache offload 适配，不能生成 DFlash speculative config。
- MemCache 部署归属：Wings 可以生成 `mmc_local.conf`、`mmc_meta.conf`、`start_memcache_master.sh` 和 engine prelude，但 MetaService 是否由 sidecar、独立服务还是启动脚本拉起，需要部署层明确。
- Pro 5000 卡型 token：`G6550+RTX PRO 5000 * 8` 与 `TokenBox RTX PRO 5000 * 8` 必须统一到当前项目可识别的标准 token，再补白名单；不要在每个白名单行复制多种拼写。
- Pro 5000 exact model id：Qwen / MiniMax 场景应优先使用真实 `model_name` / `model_path`，不要从显示名称派生宽松别名；精度后缀只有开源模型名本身包含时才进入 token。
- Native offload 容量：行 20、21、22、23、24、49 不能从样例命令继承固定容量；容量必须来自页面或 `KV_MEM_OFFLOAD_SIZE`，缺失时应关闭或保持 `auto` 规则。
- GLM4.7 拓扑：行 17、42 以及额外双机场景要复核当前代码对 `data_parallel_size`、`tensor_parallel_size`、`data_parallel_size_local` 的推导是否与实际卡数一致；不以 Excel 单条命令作为唯一真相。
- 双机网络信息：行 41 和 GLM4.7 双机的 `node_rank`、master IP、网卡名、RPC 端口属于部署输入，文档只能给适配模板，不能硬编码生产地址。
- 行 18 证据不足：Q 列为 `无优化` 且 O 列没有可执行命令，任何 spec / sparse / offload 都需要新证据或产品确认后才能加入。

### 验收建议

- 每个场景至少做一次 dry-run，检查最终 `start_command.sh`，不要只看 defaults 或白名单是否存在。
- 有标准命令证据的场景，优先沉淀为 scenario JSON，再用 `wings_control/docs/启动命令标准化校验/startup_command_standard.py verify` 调用 `build_launcher_plan()` 做同场景语义比对。`comparison.md` 中允许命令顺序、JSON key 顺序、host 包装方式等非语义差异；关键参数缺失或错误必须失败。
- 每个完成场景至少保留以下证据：scenario JSON、标准命令或来源说明、`actual_start_command.sh`、`merged_params.json`、`comparison.md`、`advanced_features.json` 片段、`/v1/startup/accel` 样例或等价状态输出。
- 重点核对最终命令中的 `--speculative-config`、`--kv-transfer-config`、`--kv-offloading-backend`、`--kv-offloading-size`、`--hf-overrides`、TP/DP 和 parser 参数。
- 同步核对 `merged_params.json`、`advanced_features.json`、`/v1/startup/accel` 返回值和日志中的最终启动命令，确保请求态和有效态没有混淆。
- 对 MemCache 场景，必须同时验证 `mmc_local.conf`、`mmc_meta.conf`、`MMC_LOCAL_CONFIG_PATH`、`start_memcache_master.sh`、ConfigStore readiness 和 fallback cleanup。
- 对 native offload 场景，确认没有残留 LMCache / MemCache connector；对 LMCache / MemCache 场景，确认没有误走 native backend。
- 适配完成标准：16 个场景都能明确落到“已支持并通过语义验收”“仅 baseline，不启用优化”“缺少外部证据/依赖，暂不实现”三类之一，不能留下只有命令片段但无当前项目结论的状态。

两种模式的判定逻辑：

- `features_off`：关闭 function call、thinking、spec、sparse、kv offload、LMCache 等开关；对齐 Excel P 列 baseline 命令中的基础字段，并强制最终命令不得出现 `--speculative-config`、`--kv-offloading-*`、`--kv-transfer-config`、`--hf-overrides` 或 LMCache/MemCache 相关环境变量。
- `features_on`：打开 function call、thinking、spec、sparse、kv offload、LMCache 等开关，但不额外传 offload 容量或 draft path；优先对齐 Excel Q 列优化命令，Q 列没有可执行命令时回退到 P 列。这里重点验证白名单、defaults、adapter 是否能在“仅输入/输出长度 + 特性开关”的条件下生成 Excel 中的关键特性字段。
- 所有 case 都比较最终 `start_command.sh` 中的结构化字段，而不是比较整段命令文本；JSON 参数会解析后比较，避免 key 顺序影响结论。
- 双机场景按节点分别检查，rank0 / rank1 都必须能产出预期 engine 命令；非 0 rank 还需要符合 `--headless` 行为。

本次基线结论：

- 脚本和项目 dry-run 链路可用，已能生成字段级差异，不再只是静态文档分析。
- 16 个来源场景在当前代码下还没有任何一个能在上述严格口径下通过；其中 30 个失败 case 属于“已有基础上简单修改”场景，12 个失败 case 属于“需要重新适配”场景。
- GLM4.7 910B/910C 单机与额外双机场景的主要 blocker 是 `speculative-config` 仍为 suffix，而文档目标是 `{"method":"mtp","num_speculative_tokens":3}`；双机 rank1 还缺可执行 engine 命令。
- DeepSeek-Coder-V2-Instruct 的 dry-run 能生成当前项目命令，但 Excel 中没有可解析参考命令，因此报告以 `reference_parse` 失败标出；后续应先把本文第五类补充命令沉淀为标准参考，再纳入同一校验。
- Pro 5000 场景在“仅输入/输出长度 + 特性开关”的条件下无法推出 Excel 中固定 offload 容量，后续如果产品要求对齐 `20/40/200GB` 等字段，必须把容量作为显式输入或场景 profile/defaults，而不是只依赖特性开关。
